# 1. Контекст продукта

Проект должен стать техническим ядром медиа «Кооператив Юг».

Позиционирование медиа:

«Инженерный взгляд на деньги и капитал».

Главная идея продукта:

Не просто показывать стоимость портфеля и текущую доходность, а раскладывать финансовые инструменты на:

* будущие денежные потоки;
* подтверждённые и прогнозные выплаты;
* оферты;
* амортизации;
* погашения;
* сценарии изменения купонов;
* важные даты;
* источники данных;
* неопределённость расчётов.

Продукт должен поддерживать три сценария использования.

## Сценарий A. Личный аналитический инструмент

Пользователь подключает read-only токен T-Invest и получает анализ своего портфеля.

## Сценарий B. Двигатель контента

Система создаёт машиночитаемые данные, таблицы и графики, которые можно использовать в:

* Telegram-постах;
* YouTube-видео;
* Instagram-карточках;
* Shorts;
* исследованиях «Кооператива Юг».

## Сценарий C. Публичный демонстрационный инструмент

Человек может запустить программу без брокерского токена на демонстрационном портфеле и понять возможности продукта.

---

# 2. Что продукт не должен обещать

На текущем этапе продукт не является:

* инвестиционным советником;
* торговым роботом;
* сервисом выставления заявок;
* системой автоматического управления капиталом;
* полноценным анализатором кредитоспособности эмитента;
* гарантией получения будущих выплат.

Приложение должно оставаться read-only.

Запрещено:

* размещать или отменять торговые заявки;
* запрашивать токен с торговыми правами;
* сохранять токен в базе данных;
* публиковать номера брокерских счетов;
* включать личные данные в демонстрационные отчёты;
* выдавать прогноз за подтверждённый факт.

---

# 3. Главный принцип данных

Каждая финансовая величина должна иметь статус происхождения.

Минимальные статусы:

* `actual` — выплата или параметр подтверждены источником;
* `forecast` — величина рассчитана по формуле или сценарию;
* `estimated` — приблизительная оценка;
* `unknown` — данных недостаточно;
* `manual` — значение внесено пользователем.

Нельзя молча заменять отсутствующее значение нулём.

Нельзя смешивать подтверждённые и прогнозные выплаты без маркировки.

Для каждого расчёта, где это возможно, сохраняй:

* источник;
* дату получения данных;
* дату расчёта;
* используемый сценарий;
* формулу;
* валюту;
* допущения;
* статус качества данных.

# Этап 2. Одна архитектура и одна CLI-команда

## Проблема

Сейчас в проекте существуют две параллельные структуры:

* `invest_bonds`;
* `app`.

Публичная команда и новые команды не должны вести в разные реализации.

## Цель

Создать одну каноническую точку входа:

```bash
south-invest
```

Желаемый интерфейс:

```bash
south-invest portfolio snapshot
south-invest cashflow
south-invest floaters forecast
south-invest floaters scenarios
south-invest offers
south-invest report
south-invest demo report
```

Допустима более простая структура, если она лучше соответствует текущему коду, но команды должны быть последовательными и предсказуемыми.

## Архитектурные требования

Раздели систему минимум на следующие слои.

### Domain

Чистые финансовые сущности и правила:

* позиция;
* финансовый инструмент;
* денежный поток;
* купон;
* погашение;
* амортизация;
* оферта;
* сценарий ставки;
* прогнозная выплата;
* статус источника.

Domain не должен:

* импортировать Rich;
* печатать в консоль;
* читать `.env`;
* обращаться к T-Invest;
* работать напрямую с SQLite.

### Application

Сценарии использования:

* загрузить портфель;
* построить cashflow;
* рассчитать флоатеры;
* найти оферты;
* сформировать отчёт;
* запустить демо.

Application должен работать через интерфейсы, а не через конкретный SDK.

### Integrations

Интеграция с T-Invest:

* получение счетов;
* получение позиций;
* получение инструментов;
* получение купонов;
* получение событий;
* получение валютных курсов.

Вся специфическая логика T-Invest должна быть изолирована здесь.

### Infrastructure

* SQLite;
* YAML;
* файловый экспорт;
* настройки;
* логирование.

### Interfaces

* CLI;
* в будущем API или Telegram-бот.

### Reporting

* преобразование доменных результатов в отчётные модели;
* table renderer;
* JSON renderer;
* CSV renderer;
* HTML renderer.

## Миграция

Не удаляй `invest_bonds` сразу.

Рекомендуемый порядок:

1. Определить канонические модели.
2. Перенести отсутствующую функциональность из `invest_bonds`.
3. Перевести новую CLI на канонический application layer.
4. Добавить deprecated wrapper для старой команды.
5. Обновить тесты.
6. Только после подтверждения удалить устаревший код.

Старая команда может временно выводить предупреждение:

> `t-invest-bonds` is deprecated. Use `south-invest`.

## Разделение большого CLI-файла

Не держи всю логику в одном `cli.py`.

Предлагаемая структура:

```text
interfaces/
  cli/
    main.py
    parser.py
    commands/
      portfolio.py
      cashflow.py
      floaters.py
      offers.py
      report.py
      demo.py
    renderers/
      table.py
      errors.py
```

Команда CLI должна:

1. разобрать аргументы;
2. вызвать application service;
3. передать результат renderer;
4. вернуть exit code.

Она не должна сама рассчитывать финансовые показатели.

## Критерии готовности

* Все основные возможности доступны через `south-invest`.
* Бизнес-логика не зависит от Rich.
* T-Invest SDK изолирован.
* Старые тесты проходят.
* Добавлены тесты новой CLI.
* `python -m app.cli` больше не является основным способом запуска.
* Есть инструкция миграции.

# Этап 3. Машиночитаемый JSON и CSV

## Зачем это нужно

Терминальная таблица удобна человеку, но непригодна как интерфейс для:

* контент-агентов;
* Telegram-бота;
* сайта;
* графиков;
* автоматической публикации;
* сравнения отчётов.

Каждая аналитическая команда должна уметь отдавать структурированные данные.

## Общий интерфейс

Добавь параметры:

```bash
--format table
--format json
--format csv
--output PATH
```

Значение по умолчанию:

```bash
--format table
```

Примеры:

```bash
south-invest cashflow --months 12 --format json
south-invest cashflow --months 12 --format csv --output cashflow.csv
south-invest offers --format json --output offers.json
south-invest portfolio snapshot --format json
```

## Требования к JSON

JSON должен быть стабильным и версионируемым.

Пример верхнего уровня:

```json
{
  "schema_version": "1.0",
  "report_type": "cashflow",
  "generated_at": "2026-07-06T12:00:00Z",
  "as_of": "2026-07-06",
  "report_currency": "RUB",
  "accounts": [],
  "summary": {},
  "items": [],
  "warnings": [],
  "data_quality": {}
}
```

## Правила сериализации

* Даты — ISO 8601.
* `Decimal` — строкой, а не float.
* Валюта — отдельным полем.
* Денежное значение — объект или унифицированная модель.
* Все прогнозные значения содержат `source_status`.
* Все сценарные значения содержат `scenario_id`.
* Не включать токен.
* Не включать номер счёта без явного разрешения.
* Для аккаунта использовать анонимный локальный идентификатор или флаг `--include-account-id`.

Пример денежного значения:

```json
{
  "amount": "1250.45",
  "currency": "RUB"
}
```

Пример прогнозной выплаты:

```json
{
  "instrument_name": "Demo Floater",
  "isin": "DEMO000002",
  "payment_date": "2026-09-15",
  "payment_type": "coupon",
  "amount_per_unit": {
    "amount": "42.10",
    "currency": "RUB"
  },
  "quantity": "10",
  "total": {
    "amount": "421.00",
    "currency": "RUB"
  },
  "source_status": "forecast",
  "scenario_id": "base-2026-07",
  "assumptions": [
    "key_rate=16.00",
    "spread=2.00"
  ]
}
```

## Требования к CSV

CSV должен:

* быть UTF-8;
* иметь стабильный порядок колонок;
* использовать ISO-даты;
* не смешивать несколько типов таблиц в одном файле;
* сохранять исходную валюту;
* иметь отдельные колонки для статуса и сценария.

Если одна команда создаёт несколько таблиц, при передаче директории создавай несколько файлов:

```text
cashflow_monthly.csv
cashflow_events.csv
cashflow_maturities.csv
```

## Архитектура renderer

Бизнес-логика должна возвращать отчётную модель.

Renderer преобразует её в:

* Rich table;
* JSON;
* CSV;
* позднее HTML.

Не создавай отдельные расчёты для каждого формата.

## Ошибки

Ошибки при JSON-режиме также желательно возвращать структурированно:

```json
{
  "error": {
    "code": "CONFIG_MISSING_TOKEN",
    "message": "INVEST_TOKEN is required for this command"
  }
}
```

## Тесты

Добавь тесты:

* валидность JSON;
* стабильность ключей;
* сериализация Decimal;
* сериализация дат;
* отсутствие токена;
* CSV-заголовки;
* корректная обработка `unknown`;
* одинаковые итоговые суммы в table, JSON и CSV.

## Критерии готовности

* Cashflow, оферты, портфель и флоатеры экспортируются в JSON.
* Основные подробные таблицы экспортируются в CSV.
* Расчёты не дублируются.
* JSON пригоден для чтения другим ИИ-агентом.
* Формат имеет `schema_version`.

# Этап 4. Демонстрационный портфель без токена

## Цель

```bash

south-invest demo cashflow --months 12
south-invest demo offers
south-invest demo floaters
south-invest demo report
```

Либо:

```bash
south-invest report --demo
```

Выбери наиболее последовательный интерфейс.

## Состав демонстрационного портфеля

Используй синтетические инструменты, чтобы данные не устаревали и не воспринимались как инвестиционная рекомендация.

Минимальный набор:

### 1. Государственная облигация с фиксированным купоном

Показывает:

* номинал;
* фиксированные купоны;
* погашение;
* регулярный cashflow.

### 2. Корпоративная облигация с фиксированным купоном

Показывает отличие эмитента, но не требует полноценного кредитного анализа.

### 3. Облигация с плавающим купоном

Показывает:

* связь с ключевой ставкой;
* spread;
* сценарии;
* `actual` и `forecast`.

### 4. Амортизируемая облигация

Показывает:

* частичный возврат номинала;
* уменьшение будущих купонов;
* отличие амортизации от дохода.

### 5. Облигация с офертой

Показывает:

* дату оферты;
* срок подачи поручения;
* предупреждение;
* риск ошибочно ориентироваться только на дату погашения.

### 6. Валютная облигация

Показывает:

* исходную валюту выплаты;
* конвертацию в валюту отчёта;
* используемый валютный курс;
* валютный риск.

Дополнительно можно добавить акцию с дивидендом для демонстрации общего portfolio cashflow, если это уже поддерживается текущей моделью.

## Данные демо

Создай fixture или provider:

```text
demo/
  portfolio.json
  instruments.json
  events.json
  scenarios.yaml
```

Или Python factory с детерминированными моделями.

Демонстрационные данные должны:

* быть детерминированными;
* работать без сети;
* иметь понятную дату `as_of`;
* не зависеть от текущей даты системы без необходимости;
* содержать все нужные типы событий;
* иметь комментарии и документацию.

Например:

```bash
south-invest demo report --as-of 2026-07-01
```

должен всегда давать одинаковый результат.

## Приватность

Демо не должно использовать:

* настоящие account ID;
* реальные позиции владельца репозитория;
* значения из локальной SQLite;
* переменные окружения;
* токен.

## Документация

Добавь в README раздел:

> Try without a broker account

Он должен содержать одну команду, после которой пользователь сразу видит результат.

## Применение для «Кооператива Юг»

Демо-портфель должен быть пригоден как учебный материал.

На его основе можно показать:

* как флоатеры реагируют на ставку;
* почему оферта требует действия;
* почему выплаты распределены неравномерно;
* чем прогноз отличается от подтверждённой выплаты.

## Тесты

Добавь snapshot или fixture tests:

* одинаковый результат для одной даты;
* полный набор типов событий;
* отсутствие сетевых вызовов;
* отсутствие токена;
* корректные суммы;
* корректные статусы `actual`, `forecast`, `unknown`.

## Критерии готовности

* Демо запускается одной командой.
* Не нужен `.env`.
* Не нужен интернет.
* Покрыты основные типы облигаций.

## Графики

### 1. Monthly cashflow

Stacked bar chart:

* фиксированные купоны;
* плавающие купоны;
* дивиденды;
* амортизации;
* погашения.

Зритель должен понять, из чего состоит cashflow каждого месяца.

### 2. Cumulative cashflow

Линия накопленного денежного потока.

Важно подписать, что это не прибыль портфеля, если в сумму входят возвраты номинала.

### 3. Payment structure

Распределение выплат по типам.

Не называй амортизации и погашения доходом.

### 4. Offers timeline

Ближайшие оферты и даты, требующие внимания.

### 5. Floating scenarios

Сравнение:

* текущего сценария;
* базового;
* стрессового;
* либо ± изменение ставки.

На графике должно быть видно, какие значения прогнозные.

## Принцип интеллектуальной честности

В отчёте визуально различай:

* подтверждённые выплаты;
* прогнозы;
* неизвестные значения.

Например:

* сплошная заливка — actual;
* штриховка — forecast;
* пустое или серое поле — unknown.

Не создавай ложное ощущение точности.

## Анонимизация

Добавь:

```bash
--anonymize
```

При включённом режиме:

* скрывать account ID;
* не показывать внутренние идентификаторы;
* опционально скрывать количество активов;
* позволять заменить названия инструментов на псевдонимы;
* не включать среднюю цену покупки, если пользователь этого не хочет.

Это необходимо для публикации скриншотов в «Кооперативе Юг».

# Этап 6. Версионирование формул и сценариев

Формулы флоатеров и сценарии ставок не должны быть безымянными YAML-файлами, происхождение которых невозможно установить.

## Для сценария сохраняй

* `id`;
* название;
* описание;
* дату создания;
* дату обновления;
* автора;
* источник;
* валюту или рынок;
* временной диапазон;
* значения ставки;
* допущения;
* статус: `draft`, `active`, `archived`.

Пример:

```yaml
id: cb-key-rate-base-2026-07
name: Base key-rate scenario
status: active
created_at: 2026-07-06
updated_at: 2026-07-06
author: Cooperative South
source:
  title: Manual editorial scenario
  url: null
assumptions:
  - Gradual easing
rates:
  2026-07: 18.0
  2026-08: 18.0
  2026-09: 17.0
```

## Для формулы флоатера сохраняй

* ISIN;
* название;
* базовый индекс;
* spread;
* day count;
* lag;
* cap;
* floor;
* источник;
* дату проверки;
* комментарий;
* статус качества данных.

Если точная формула неизвестна, инструмент должен вернуть `unknown`, а не использовать похожую формулу другого выпуска.
